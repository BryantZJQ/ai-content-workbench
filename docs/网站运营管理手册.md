## 三、卡密管理（核心运营动作）

### 3.1 生成卡密

```powershell
cd F:\00_tips\content-pipeline-web

# 生成试用卡（3次用完即废，用于引流）
python key_manager.py --plan demo --count 10

# 生成体验卡（3天有效，用于深度体验）
python key_manager.py --plan trial --count 5

# 生成月卡（30天，用于售卖）
python key_manager.py --plan monthly --count 5

# 查看库存
python key_manager.py --list
```

### 3.2 同步到云端

生成卡密后，**必须同步到 Streamlit Cloud Secrets**，否则线上无法验证：

1. 打开 https://share.streamlit.io
2. 找到 **ai-content-workbench** → 点 **⋮** → **Settings** → **Secrets**
3. 在已有内容**末尾追加**新卡密：

```toml
# === 新增试用卡 ===
[keys.ACP-XXXX-XXXX-XXXX]
plan = "demo"
status = "unused"

[keys.ACP-YYYY-YYYY-YYYY]
plan = "monthly"
status = "unused"
```

4. 点 **Save**，等30秒自动生效

### 3.3 卡密库存管理策略

**建议常备库存：**

| 类型           | 常备数量 | 补货阈值     | 用途            |
| -------------- | -------- | ------------ | --------------- |
| 试用卡 (demo)  | 20个     | 剩余<5时补货 | 大量发放引流    |
| 体验卡 (trial) | 10个     | 剩余<3时补货 | 公众号/社群福利 |
| 月卡 (monthly) | 10个     | 剩余<3时补货 | 主力变现        |
| 年卡 (yearly)  | 3个      | 按需生成     | 高价值用户      |

### 3.4 卡密发放台账

**强烈建议维护一个Excel/腾讯文档，记录每张卡密的去向：**

| 卡密         | 类型   | 发放对象     | 发放渠道   | 发放日期   | 是否付费 | 金额 |
| ------------ | ------ | ------------ | ---------- | ---------- | -------- | ---- |
| ACP-XXXX-... | 试用卡 | 小红书@用户A | 小红书私信 | 2026-04-03 | 否       | -    |
| ACP-YYYY-... | 月卡   | 微信用户B    | 微信转账   | 2026-04-03 | 是       | ¥29  |

**为什么这很重要：**

- 知道哪些卡已发、哪些未发
- 追踪付费转化率（试用→付费的比例）
- 用户问题时能快速定位
- 月底算账需要

## 四、成本控制（最关键的运营风险）

### 4.1 DeepSeek API 计费规则

DeepSeek Chat 模型定价（2026年参考）：

| 计费项               | 价格           |
| -------------------- | -------------- |
| 输入 (input tokens)  | ~¥1/百万tokens |
| 输出 (output tokens) | ~¥2/百万tokens |
| 缓存命中 (cache hit) | 更低           |

### 4.2 每个功能的大致成本

| 功能       | 单次调用token量 | 估算成本/次 | 说明           |
| ---------- | --------------- | ----------- | -------------- |
| 热搜抓取   | 0（不用API）    | ¥0          | 纯爬虫         |
| 智能选题   | ~3000 tokens    | ~¥0.01      | 打分排序       |
| 脚本生成   | ~4000 tokens    | ~¥0.015     | 单次生成       |
| 分镜提示词 | ~8000 tokens    | ~¥0.03      | 多镜头输出     |
| 一键出片   | ~15000 tokens   | ~¥0.05      | 选题+脚本+分镜 |
| 爆款拆解   | ~4000 tokens    | ~¥0.015     | 分析输出       |
| 标题A/B    | ~3000 tokens    | ~¥0.01      | 多标题生成     |
| 脚本诊断   | ~4000 tokens    | ~¥0.015     | 诊断报告       |

### 4.3 成本预警线

**假设月卡用户每天用10次，以一键出片为主：**

- 单用户日成本：10 × ¥0.05 = ¥0.5
- 单用户月成本：¥0.5 × 30 = **¥15**
- 月卡售价 ¥29，**毛利 ¥14（毛利率48%）**

**假设试用卡用户3次全用一键出片：**

- 成本：3 × ¥0.05 = **¥0.15**（几乎可以忽略）

**危险场景：**

- 如果有用户恶意高频调用（1分钟调100次）→ 目前有每日次数限制保护
- 如果同时在线用户暴涨 → DeepSeek API可能触发限流

### 4.4 充值建议

| 阶段       | 建议充值         | 预期支撑          |
| ---------- | ---------------- | ----------------- |
| 初期测试   | ¥10              | 约200-1000次调用  |
| 正式运营   | ¥50              | 约1000-5000次调用 |
| 用户增长后 | 按月消耗×1.5倍充 | 留安全余量        |

### 4.5 API余额检查

1. 登录 https://platform.deepseek.com
2. 查看 **用量统计** 和 **账户余额**
3. **设置低余额提醒**（如果平台支持）
4. 余额低于 ¥10 时立即充值

**⚠️ API余额归零 = 网站所有AI功能瘫痪！这是最大运营风险。**

## 五、故障处理

### 5.1 常见故障速查表

| 现象             | 可能原因              | 解决方案                                         |
| ---------------- | --------------------- | ------------------------------------------------ |
| 网站打不开       | Streamlit Cloud宕机   | 等待恢复（通常几分钟），或查 status.streamlit.io |
| 所有AI功能报错   | DeepSeek API余额为0   | 登录 platform.deepseek.com 充值                  |
| 用户说卡密无效   | 卡密没同步到Secrets   | 检查Streamlit Cloud Secrets里有没有这个卡密      |
| 卡密显示"已过期" | 正常过期              | 给用户发新卡密                                   |
| 热搜加载空白     | 热搜源网站接口变化    | 暂时无法修复，不影响其他功能                     |
| 脚本生成很慢     | DeepSeek API排队      | 正常，高峰期可能30-60秒                          |
| 页面样式错乱     | Streamlit版本更新导致 | 检查CSS选择器是否匹配                            |
| git push失败     | 网络问题（常见）      | 换个时间重试，或挂VPN                            |

### 5.2 紧急恢复操作

**场景：API Key泄露**

1. 立即登录 platform.deepseek.com → 重新生成API Key
2. 更新 Streamlit Cloud Secrets 里的 `DEEPSEEK_API_KEY`
3. 同时更新本地环境变量

**场景：网站被人恶意刷量**

1. 在 Streamlit Cloud Secrets 里把对应卡密改为 `status = "expired"`
2. 或直接删除该卡密条目
3. Save 后30秒生效

**场景：代码推送后网站崩溃**

```powershell
# 回退到上一个版本
cd F:\00_tips\content-pipeline-web
git log -n 5 --oneline    # 查看最近5次提交
git revert HEAD            # 撤销最后一次提交
git push origin main       # 推送，Streamlit自动回滚
```

## 七、Supabase 数据库管理

### 7.1 它是干什么的？

Supabase 在本项目中负责**云端持久化卡密使用次数**。具体来说：

- 用户每用一次付费功能 → `cloud_db.py` 向 Supabase 写入 `+1`
- 验证卡密时 → 从 Supabase 读取真实已用次数
- **核心作用**：让试用卡的3次限制在多设备/多浏览器/清缓存后依然准确

### 7.2 免费额度与限制

| 项目       | Free Plan 额度 | 你的实际用量         | 是否够用    |
| ---------- | -------------- | -------------------- | ----------- |
| 数据库容量 | 500 MB         | < 1 MB（纯文本记录） | ✅ 绰绰有余 |
| API请求数  | 无硬限制       | 每次功能调用1次      | ✅ 完全够用 |
| 存储       | 1 GB           | 不使用               | ✅          |
| 带宽       | 5 GB/月        | 极少                 | ✅          |
| 并发连接   | 不限           | 极少                 | ✅          |

**结论：以当前和可预见的用量，Supabase Free Plan 完全免费，无需付费升级。**

### 7.3 ⚠️ 唯一需要注意的事

> **Free Plan 项目如果连续 7 天没有任何 API 请求，会被自动暂停（Paused）。**

- 暂停后**数据不会丢失**
- 但暂停期间用量记录功能不可用
- 需要手动去 Dashboard 点 "Restore project" 恢复（约1-2分钟）

**什么情况会触发暂停：**

- 你的网站连续7天没有任何用户使用过付费功能（热搜是免费的，不触发数据库请求）
- 初期用户少的时候最容易触发

**如何避免：**

- 只要每周有1个人使用过任意付费功能（选题/脚本/分镜等），就不会暂停
- 如果实在没用户，你自己每周上去用一次也行

### 7.4 被暂停了怎么恢复？

1. 打开 [Supabase Dashboard](https://supabase.com/dashboard/project/yrvvhdcrivzyrzioqdjd)
2. 看到提示 "Project is paused"
3. 点击 **"Restore project"**
4. 等待 1-2 分钟恢复完成
5. 网站自动恢复正常，无需其他操作

### 7.5 如果 Supabase 挂了，网站会怎样？

| 功能              | 影响        | 说明                                    |
| ----------------- | ----------- | --------------------------------------- |
| 🔥 热搜榜单       | ✅ 正常     | 不依赖数据库                            |
| 🎯 智能选题       | ✅ 正常     | 不依赖数据库                            |
| 📝 脚本生成       | ✅ 正常     | 不依赖数据库                            |

| 🔍 爆款拆解       | ✅ 正常     | 不依赖数据库                            |
| 🏷️ 标题优化       | ✅ 正常     | 不依赖数据库                            |
| 📊 脚本诊断       | ✅ 正常     | 不依赖数据库                            |

## 九、安全事项

### 9.1 需要保密的信息

| 信息             | 泄露后果            | 存放位置                      |
| ---------------- | ------------------- | ----------------------------- |
| DeepSeek API Key | 别人盗刷你的API余额 | Streamlit Secrets + 环境变量  |
| Supabase Key     | 数据库被篡改        | Streamlit Secrets             |
| 未发放的卡密     | 别人免费使用        | 本地 data/keys.json + Secrets |
| GitHub密码       | 代码被篡改          | GitHub账号设置                |

### 9.2 安全检查清单

- [ ] API Key 不在代码文件中出现（用 Secrets 管理）
- [ ] 卡密不在 GitHub 代码中出现
- [ ] `data/` 目录已在 `.gitignore` 中
- [ ] GitHub 账号开启了两步验证
- [ ] 定期检查 DeepSeek 用量是否异常

### 9.3 卡密安全

当前卡密通过URL参数（base64编码）持久化。这不是加密，只是编码，有经验的人可以解码。但对普通用户足够安全，因为：

- 每个卡密有独立的用量限制
- 试用卡只有3次，刷了也无所谓
- 月卡/年卡有每日上限保护
